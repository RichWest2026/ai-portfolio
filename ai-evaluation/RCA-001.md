# RCA-001: LLM Character-Counting Failure with Compounding Confidence Error

**Document Type:** Root Cause Analysis (RCA)
**Classification:** AI Behavioral Failure — Factual Hallucination / Overconfidence
**Severity:** Medium
**Model:** Claude (Sonnet series, claude.ai interface)
**Date of Incident:** May 2026
**Date of RCA:** June 2026
**Author:** R. West
**Status:** Closed — Root Cause Confirmed

---

## 1. Executive Summary

During an extended Claude conversation, the model was asked to analyze an email containing a stylistically elongated word ("sooooooooooooo"). Claude independently counted the repeated characters, derived a count of 8, coined a shorthand term ("eight-o reply") based on that count, and used the term repeatedly throughout the conversation as an established reference point. When the user later asked what "eight-o" meant, Claude explained the term and defended the count — at which point the user identified the correct count as 13, a discrepancy of 5 characters (38% error).

This incident illustrates three compounding failure modes: (1) inaccurate character counting due to tokenization architecture, (2) false confidence in an unverified factual claim, and (3) terminological entrenchment — building persistent vocabulary around an incorrect foundation without flagging uncertainty at any stage.

## 3. Failure Mode Analysis

### 3.1 Primary Failure — Character-Counting Inaccuracy

**What happened:** Claude miscounted the number of repeated characters in a token sequence it had direct access to in context.

**Root cause:** Large language models do not process text as discrete characters. Input is converted to tokens — subword units that may compress multiple characters into a single token. The string "sooooooooooooo" (1 s + 13 o's) is likely represented as 2–4 tokens depending on the tokenizer's vocabulary, not as 14 individual character units. When the model attempts to "count" characters, it is reasoning over token representations, not performing a direct character-by-character enumeration. This is a known and documented architectural limitation of transformer-based LLMs.

**Contributing factor:** The model was not prompted to count. It self-initiated the count as part of an analytical observation, without being asked and without flagging that character-level counting is an unreliable operation for this architecture.

**Expected behavior:** Either (a) produce the correct count, or (b) hedge explicitly — e.g., "roughly a dozen o's" — given that precise character enumeration is an architecturally weak operation.

**Actual behavior:** Produced a specific, confident count (8) without qualification.

---

### 3.2 Secondary Failure — Overconfidence and Terminological Entrenchment

**What happened:** Based on the unchecked count of 8, Claude coined the term "eight-o reply" and used it as a stable, named reference across multiple subsequent exchanges. The term was deployed with full confidence — no hedging, no acknowledgment that the underlying count was an estimate.

**Root cause:** Once a model produces an output and it is not immediately corrected, that output tends to be treated as established context in subsequent turns. The term "eight-o" became part of the working vocabulary of the conversation, creating a form of self-referential reinforcement. The model had no internal mechanism to flag that the foundational fact underpinning the term was unverified.

**Contributing factor:** The term was used in a casual, stylistically confident register ("the eight-o email," "the eight-o reply"), which signals certainty to the user and reduces the likelihood of challenge. Users are less likely to question something framed as an informal nickname than something framed as a data point.

**Expected behavior:** Either verify the count precisely before coining terminology around it, or use non-specific language ("the stretched-out reply," "the long-o email") that does not embed an unverified number.

**Actual behavior:** Coined precise terminology, repeated it, and reaffirmed it when asked to explain it — at which point the error became explicit.

---

### 3.3 Tertiary Failure — Graceful Recovery Masking Underlying Unreliability

**What happened:** When the user corrected the count from 8 to 13, Claude accepted the correction immediately and pivoted to: *"Thirteen is even more deliberate than eight — which strengthens the read."*

**Why this is a failure mode, not a success:** The recovery was smooth, reasonable, and conversationally appropriate. However, it also demonstrates that the model's analytical narrative (the significance of the o-count as a signal of emotional affect) was post-hoc and flexible — it would have supported 8 just as confidently as it supported 13. This reveals that the specific number was not actually load-bearing in the analysis, despite being presented as if it were.

**Implication:** The model's ability to recover gracefully from factual errors can mask the unreliability that produced the error in the first place. A user who does not catch the original error receives an analysis built on a false foundation, stated with full confidence, with no visible seam.

## 5. Impact Assessment

**Severity in this instance:** Low-to-Medium. The specific count (8 vs. 13) was not materially consequential to the underlying advice. The conversation's core recommendation — that an unusual number of repeated characters signals elevated emotional register from a senior professional — was valid regardless of the precise count.

**Severity in analogous scenarios:** High. The same failure pattern (confident character/token counting deployed in a named construct without uncertainty flagging) applied to higher-stakes contexts — contract clause word counts, data field character limits, regulatory reference numbers, serial numbers, dosage specifications — could produce consequential errors with no visible warning. The model's confidence register does not reliably signal the accuracy of its character-level counting.

---

## 6. Detection Method

**How the error was caught:** User corrected the model directly, having independently counted the characters before transcribing the source email into the conversation. The user noted: *"It took me a second to count them out before I transcribed the email here so I remember."*

**Detection reliability:** Human review only. The error was not self-detected by the model, not flagged by any uncertainty signal, and would not have been caught without the user's independent verification. There is no automated detection path for this failure class in the current interface.

## 7. Recommendations

### For Users Working with Claude

| Recommendation | Rationale |
|----------------|-----------|
| Do not rely on Claude for precise character, word, or token counts without independent verification | Tokenization architecture makes this structurally unreliable |
| Treat named shorthands coined by Claude as potentially encoding unverified facts | Terms like "the eight-o email" embed a specific claim; verify the claim before the term becomes conversational convention |
| When precision matters, explicitly prompt Claude to flag uncertainty: "Count these characters and tell me how confident you are" | Eliciting uncertainty acknowledgment improves calibration |
| For character-counting tasks, request that Claude output the string being counted alongside the count, enabling easy visual verification | Transparency check that requires minimal user effort |

### For AI Developers / Evaluators

| Recommendation | Rationale |
|----------------|-----------|
| Add explicit uncertainty signaling for character-level counting operations | Model should hedge when performing operations that are architecturally unreliable |
| Evaluate for "terminological entrenchment" — cases where an error becomes embedded in coined vocabulary | This failure class is harder to detect than single-turn errors |
| Test graceful recovery behavior for whether it accurately signals the reliability of the original claim | Smooth recovery can mask structural unreliability |
| Consider character-counting as a benchmark category for LLM factual reliability evaluation | Failure is reproducible, verifiable, and architecturally explained |
